## Workflow ที่ต้องยึดทุกครั้ง

1. เช็ก HEAD ล่าสุดของ `feature/retail-pos` ก่อนแก้ไขทุกครั้ง
2. อ่าน `PROJECT_CONTEXT.md` ก่อนเริ่มงาน
3. แก้แบบเล็กและเฉพาะจุด เพื่อลดผลกระทบกับระบบที่ใช้งานได้แล้ว
4. หลังแก้ต้องแจ้ง HEAD ล่าสุดและไฟล์ที่แก้
5. แจ้งคำสั่ง deploy ทุกครั้ง

```bash
git pull --rebase origin feature/retail-pos
firebase deploy --only hosting
```

## สถานะล่าสุดของระบบที่ทำไปแล้ว

### 1. Table Order / QR Order

- ลูกค้าสั่งอาหารจาก QR ของโต๊ะ
- มีตะกร้าและรายการรอบปัจจุบัน
- แสดงรายการที่โต๊ะนี้เคยสั่งแล้ว
- รองรับการสั่งหลายรอบจากโต๊ะเดียว
- ปรับ mobile spacing ระหว่างรายการอาหารกับตะกร้าแล้ว

### 2. Kitchen

- ครัวรับออเดอร์แบบ realtime
- เปลี่ยนสถานะออเดอร์ได้: รอรับออเดอร์, ครัวรับแล้ว, กำลังทำ, พร้อมเสิร์ฟ, เสิร์ฟแล้ว
- แก้ไขรายการอาหารรายตัวได้ เช่น ลดจำนวน เปลี่ยนเมนู เพิ่มหมายเหตุ
- ยกเลิกรายการรายตัวได้
- ยกเลิกทั้งออเดอร์ได้
- เพิ่มปุ่มเสิร์ฟรายตัวเฉพาะออเดอร์โต๊ะในร้าน
- ถ้ากดเสิร์ฟทั้งออเดอร์แล้ว ปุ่มเสิร์ฟรายตัวจะไม่แสดงซ้ำ
- แก้ Toast ไม่ให้ขึ้นข้อความซ้ำแบบ `เสิร์ฟแล้ว แล้ว`

### 3. Cashier

- แสดงบิลรวมตามโต๊ะ
- แสดงยอดรวมทั้งโต๊ะ
- รับชำระเงินและปิดบิลได้
- พิมพ์ใบเสร็จได้
- ดูรายละเอียดออเดอร์เดิม/รายการเดิมได้
- เพิ่มระบบเปลี่ยนโต๊ะแล้ว

### 4. Table Move

เงื่อนไขที่ทำไว้:

- โต๊ะเดิมต้องยัง active / occupied
- โต๊ะใหม่ต้องว่างเท่านั้น
- ย้ายเฉพาะออเดอร์โต๊ะที่ยังไม่ชำระ
- โต๊ะเก่าถูกปิดเป็นว่าง
- โต๊ะใหม่ถูกเปิดด้วย token เดิม
- ออเดอร์เดิมย้าย `tableCode/tableName` ไปโต๊ะใหม่

### 5. Table Close Guard

แนวคิดที่ต้องรักษาไว้:

- ห้ามปิดโต๊ะถ้ายังมีออเดอร์ค้างหรือยังไม่ชำระเงิน
- ต้องแสดงข้อความ: `ยังมีออเดอร์หรือยังไม่ได้ชำระเงิน ไม่สามารถปิดโต๊ะได้`
- เคสสำคัญที่ต้อง test ซ้ำเสมอ: เปิดโต๊ะ 4 > สั่งอาหาร > ย้ายไปโต๊ะ 2 > ยังไม่ชำระ > ต้องปิดโต๊ะ 2 ไม่ได้

## เรื่องที่ยังควรทำต่อ / ยังไม่เสร็จสมบูรณ์

### Priority 1 — POS หน้าขายหน้าร้าน

เป้าหมาย: เพิ่มระบบ POS สำหรับสั่ง/ขายที่เคาน์เตอร์ โดยไม่ต้องใช้ QR โต๊ะ

รายการที่ควรทำ:

- สร้างหน้า POS หลัก เช่น `/pos`
- เลือกเมนูลงตะกร้าได้เร็ว
- เพิ่ม/ลดจำนวนในตะกร้า
- รองรับส่วนลดระดับบิล
- รองรับภาษี/ค่าบริการ ถ้าต้องใช้ในอนาคต
- เลือกวิธีชำระเงิน: เงินสด, โอน, QR, อื่น ๆ
- รับเงินสดและคำนวณเงินทอน
- บันทึกออเดอร์ POS แยกจาก table/delivery เช่น `orderType: "pos"`
- พิมพ์ใบเสร็จหลังชำระเงิน
- ปิดออเดอร์เป็น `paid` ทันทีเมื่อรับเงิน

### Priority 2 — รายงานยอดขาย

- รายงานยอดขายรายวัน
- แยกยอดตามวิธีชำระเงิน
- แยกยอดตามช่องทาง: โต๊ะ, POS, Delivery
- สรุปจำนวนบิล / ยอดรวม / ส่วนลด / ยอดสุทธิ
- Export CSV หรือพิมพ์รายงานได้ในอนาคต

### Priority 3 — ความแข็งแรงของปิดโต๊ะ/ย้ายโต๊ะ

ยังต้อง test ซ้ำหลัง deploy:

- ย้ายโต๊ะแล้วยังไม่ชำระ ต้องปิดโต๊ะใหม่ไม่ได้
- ชำระเงินจาก cashier แล้วโต๊ะใหม่ต้องกลับเป็นว่าง
- โต๊ะเก่าหลังย้ายต้องว่างจริง
- QR โต๊ะเก่าต้องใช้งานต่อไม่ได้
- QR โต๊ะใหม่/token เดิมต้องยังสั่งต่อได้หรือไม่ ตาม requirement ที่ต้องสรุปให้ชัด

หมายเหตุ: เคสนี้เคยมี bug หลุดปิดโต๊ะได้หลังย้ายโต๊ะ จึงควรถือเป็น regression test หลักทุกครั้งที่แก้ cashier/table.

### Priority 4 — Sync version/milestone

ตอนนี้ `public/assets/js/app-info.js` ยังเก่ากว่างานล่าสุด ควรอัปเดตให้ตรงกับ milestone ปัจจุบัน เช่น

- version/build ล่าสุด
- commit ล่าสุด
- milestone ล่าสุด
- whatsNew ล่าสุด

### Priority 5 — README หลักยังเก่า

`README.md` ยังเป็น Version 1 และยังไม่สะท้อนระบบ Retail POS / tenant / delivery / kitchen item edit / table move จึงควรปรับในภายหลัง

หัวข้อที่ควรเพิ่มใน README หลัก:

- โครงสร้างระบบปัจจุบัน

## Regression Tests สำคัญ

### A. สั่งอาหารที่โต๊ะ

1. เปิด QR โต๊ะ
2. ลูกค้าสั่ง 1 รอบ
3. ครัวรับออเดอร์ > เริ่มทำ > พร้อมเสิร์ฟ
4. เสิร์ฟรายตัว
5. ถ้าเสิร์ฟครบ ต้องกลายเป็น `served`
6. แคชเชียร์คิดเงิน
7. โต๊ะกลับว่าง

### B. ย้ายโต๊ะ

1. เปิดโต๊ะ 4
2. สั่งอาหาร 1 รอบ
3. ย้ายไปโต๊ะ 2
4. ออเดอร์ต้องย้ายไปโต๊ะ 2
5. โต๊ะ 4 ต้องว่าง
6. โต๊ะ 2 ต้อง occupied
7. ยังไม่ชำระ ต้องปิดโต๊ะ 2 ไม่ได้
8. ชำระแล้ว จึงปิดโต๊ะ 2 ได้

### C. ครัวเสิร์ฟรายตัว

1. สั่งหลายรายการในออเดอร์เดียว
2. เปลี่ยนเป็นพร้อมเสิร์ฟ
3. กดเสิร์ฟรายตัวบางรายการ
4. รายการที่เสิร์ฟแล้วต้องไม่แสดงปุ่มเสิร์ฟซ้ำ
5. กดเสิร์ฟครบทุกตัวแล้ว order ต้องเป็น `served`
6. ถ้ากดเสิร์ฟทั้งออเดอร์ ปุ่มรายตัวต้องไม่แสดงซ้ำ

## ข้อควรระวัง

- อย่าแก้ไฟล์ใหญ่ถ้าไม่จำเป็น ให้เพิ่ม helper เฉพาะจุดก่อน
- ระวัง browser cache ต้อง bump query string ทุกครั้งที่แก้ JS/CSS ที่ import ใน HTML
- ฟีเจอร์ที่เกี่ยวกับโต๊ะต้องเช็กทั้ง `tableCode` และ `tableToken`
- ฟีเจอร์ที่เกี่ยวกับการชำระเงินต้องแยก `status` กับ `paymentStatus` ให้ชัด
- POS ควรใช้ `orderType: "pos"` เพื่อไม่ปนกับ `delivery` และ table order
